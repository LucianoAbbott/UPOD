1. Introduction

UPOD (Undergraduate Physics Online Database) is an online physics database, being a means of storing and retrieving information. Students no longer need to worry about searching through multiple sites to attain accurate information on physics. UPOD provides single access point for all undergraduate-level physics information.

1.1 Purpose

The purpose of this document is to describe the requirements for UPOD as a database and a website. It is an overview of the scope and functions of the project for UPOD's client, development team, and any other interested parties.

1.2 Scope

UPOD is an online physics database targeted at undergraduate students. However, anyone with internet access may use UPOD. The UPOD website connects to a database of physics information. It displays this information via a series of web pages, each page providing information on a different physics topic. The website allows all users to search the database and read pages. Moderators and administrators have the ability to create, update, and delete pages. UPOD increases the efficiency of obtaining physics information, as it eliminates the need to search multiple websites by providing a central location for all undergraduate physics topics.

1.5 Overview

This document provides a reference for the software specifications of UPOD. It begins with an introduction and goes on to describe the characteristics, functions, and design of the project. An overview of the requirements follows, with appendices and an index ending the document.

2. Overall Description

UPOD is a self-contained website based on the functionality of a physics encyclopedia. Therefore, UPOD supplies users with information they request in regards to physics. The goal is to make UPOD intuitive to the user through the use of known heristics in user interface (UI) design, and to make the website easier to use then other physics databases. To ensure the website is informative, moderators, in this case writers for the site, can add, delete and edit pages and information in the database while administrators, the manager of the moderators, can promote and revoke moderator status.

2.1 Product Perspective

The UPOD website is self contained supplying its own front-end and back-end. However, UPOD is a Laurier Service and can utilize the Single Sign On authentication service provided by idp.wlu.ca for moderator and admin login only. UPOD hosts full support for the following browsers.

  • Firefox 28
  • Chrome 21
  • Safari 6.1
  • Opera 12.1
  • MS Edge 11

UPOD's user interface (UI) is designed exclusively for desktop machines with little to no consideration for mobile usability.

2.2 Define Product Functions

Navigation
UPOD is navigable using a series of categorical links and a search bar. Categorical based navigation is based on a hierarchical tree structure implemented by assigning categories to pages.
Search
UPOD has a search algorithm to find and index key terms from the title and body of articles. Key terms are compared to the frequency of word use and if under a certain threshold are indexed.
Pages
Pages have a title, categories, sections, sub sections, formulas, and diagrams.
Moderator Functions
Moderators can create, delete and edit pages.
Administrator Functions
Administrators can promote, and revoke moderator status to and from user status.

2.3 User Characteristics

General Users
While UPOD is targeted at university students, a general user is not required to have physics education or experience. It is assumed that a general user has basic technical knowledge and can navigate a website. They are able to traverse the website and read pages without creating an account.
Moderators
A moderator has all of the privileges of a general user. They also require an account, which can be used to create, update, edit, and delete pages. Moderators should have a basic knowledge of physics and greater technical knowledge than a general user, as they have a greater degree of control over the website. Programming knowledge is not mandatory, however, if a moderator wishes to create an interactive diagram, they must know JavaScript.
Administrators
An administrator has all moderator privileges, and can add moderators and administrators to the pre-existing list of moderators and administrators. Administrators also require an account, and should have the greatest degrees of physics knowledge and technical experience.

2.4 Constraints

Due to the interactive and technical nature of the front-end, the website does not have an exclusive interface for mobile devices.

2.5 Assumptions and Dependencies

The assumption is that the general user is accessible to anyone, where as the administrators and moderators have access from idp.wlu.ca to edit, create, delete and update pages. By having access to idp.wlu.ca, it does not require UPOD to create a separate database for hashing and / or encrypting passwords.

3. Specific Requirements

3.1 External Interface Requirements

UPOD will have no external interface to rely on other than the Laurier database to cross reference login access for administrators and appointed moderators.

3.2 Functional Requirements

Each function description are divided into the following categories:
Use Case Name – A description of the function preformed.
Trigger – The action that causes the function to be performed.
Precondition – The precondition(s) that have to be meet for the function to be performed.
Basic Path – The website pages that appear or change based on the function preformed.
Alternative Path – An alternative path that the website can preform to fulfill the functions desired result.
Postcondition – The end result after the function is performed.
Exception Paths – Actions that can cause the function to preform incorrectly and how the function deals with them accordingly.
Other – Anything else that the function needs or occurs that needs clarification about its function.

Administrative Login

Use Case Name
Sign in to existing administrator account.
Trigger
Host clicks "Sign in as Administrator".
Precondition
Host is on host page of the web application.
Basic Path
From the host page, the user clicks "Sign as Administrator" and enters their Laurier username and Laurier password.
Alternative Path
None.
Postcondition
Host signed in to an existing account.
Exception Paths
If the username and password are incorrect, host is asked to resubmit credentials.
Other
A administrator account cannot be accessed by multiple computers.

Moderator Login

Use Case Name
Sign in to existing moderator account.
Trigger
Host clicks "Sign in as Moderator".
Precondition
Host is on host page of the web application.
Basic Path
From the host page, the user clicks "Sign as Moderator" and enters their Laurier username and Laurier password.
Alternative Path
None.
Postcondition
Host signed in to an existing account.
Exception Paths
If the username and password are incorrect, host is asked to resubmit credentials.
Other
A moderator account cannot be accessed by multiple computers.

Search UPOD:

Use Case Name
Search UPOD for specific topic.
Trigger
Request submitted via search bar.
Precondition
None.
Basic Path
From the host page, the user selects the search bar and is able to submit a query.
Alternative Path
User can also use drop down bar for general searches.
Postcondition
Search results page is shown containing results for the specific query.
Exception Paths
If no results are found, the user is presented with an empty results page stating that no results were found.
If the user submits a query without any input on the search bar, the user is presented with message requesting information.
Other
None.

3.3 Security

UPOD requires a login and password for moderators and administrators. This login is cross referenced with the Laurier single sign on authentication, so that professors and other administrators can edit the website, as well as appoint moderators and other administrators.
