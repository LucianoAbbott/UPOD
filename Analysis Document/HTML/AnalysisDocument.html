1. Introduction

UPOD (University Physics Online Database) is a comprehensive online database designed to organize information pertaining to fundamental physics concepts, and present them in a straightforward, efficient manner for user retrieval. UPOD reduces the need to search through multiple sites and provides users a central location for reliable data. This analysis document formalizes the parameters determined for the project in its requirements plan, and refines in greater detail and scope the objectives of UPOD. In addition to the project's objectives, project specifications are articulated with clarity and concision to better remove uncertainty concerning any of the project's faculties during the design and implementation phases. It is presented in HTML format with UML diagrams.

2. Purposes and Uses

The purpose of this document is to provide an in-depth analysis of the project beyond the initial requirements phase, as it is a basis for the design document as well as for the project. The class, use case, and sequence diagrams are used in conjunction with object-oriented design to create a starting point for project development.

3. Definitions

  • Actor - Any user (such as general users, administrators, moderators).
  • Boundary Object - Represents the system with actors which interact with other actors. Forms, windows, notices and messages are included.
  • Behaviour Diagram - A type of diagram that models a system's operations through the functions of its components.
  • Class Diagram - A structure diagram that outlines all classes, methods and attributes used in the project as well as their relationships.
  • Control Object - Coordinates boundary and entity objects. Created at the beginning of a use case, and ceases to exist afterwards.
  • Entity Object - Represents persistent, immutable information which form the core of the model. Often identified by nouns.
  • Interaction Diagram - A type of diagram that models how system elements affect and influence each other.
  • Sequence Diagram - An interaction diagram that shows how objects interact with each other in the temporal context of events.
  • Structural Diagram - A type of diagram that models the anatomy and architecture of a program.
  • UML - Unified Modeling Language. The basis for all diagrams used in this document.
  • Use Case - A method or action between two actors, connected through a communication (which is represented by a straight line).
  • Use Case Diagram - A behaviour diagram illustrated with actors and use cases, with communications between them to display their relationships.

4. Object Classification

4.1 Entity Objects

  • Administrator - A user who can create, read, update, and delete pages. Also has the ability to add and remove moderators and administrators, as well as flag and delete requests.
  • Formula - Created, read, updated, and deleted. Stored in the database.
  • Moderator - A user who can create, read, update, and delete pages. Can also flag and delete requests.
  • Page - Created, read, updated, and deleted. Stored in the database.
  • Suggestion - Added, flagged, and deleted. Stored in the database.

4.2 Boundary Objects

  • FormulaEdit - Creates, updates, and deletes formulas.
  • GraphicsEdit - Adds and deletes graphics.
  • Login - Where a moderator or administrator logs into UPOD to create, update, and delete pages.
  • PageEdit - Creates, updates, and deletes pages.
  • PageView - When a user views the physics information on a page.
  • Search - Where a user searches for pages.
  • Search Bar - Where a user types in keywords to be searched.
  • SearchResult - Where a user views the results of a search.

4.3 Control Objects

  • FormulaEditController - Coordinates the creating, updating, and deleting of formulas.
  • GraphicsEditController - Coordinates the adding and deleting graphics.
  • LoginController - Coordinates the logins of moderators and administrators.
  • PageEditController - Coordinates the creating, updating, and deleting of pages.
  • PageViewController - Coordinates which page to show.
  • SearchController - Coordinates the search function.

5. Diagrams

5.1 Use Case Diagram

  • Administrator has all the privileges of a Moderator and is able to Manage Permissions.
  • Moderator has all the privileges of a General User and is able to Manage Requests and Manage Content.
  • General User is able to Search UPOD, View Page, and Submit Requests.

6. Administrator/Moderator/General User Overview

  • Manage Permissions - The user's ability to give administrator and moderator privileges.
  • Manage Requests - The user's ability to delete or otherwise manage page comments.
  • Manage Content - The user's ability to delete or otherwise manage page content, such as diagrams, text, and images.
  • Search UPOD - The action taken by the user to search the database for specific information.
  • View Page - The action taken by the user to view a specific page on the database.
  • Submit Requests - The action taken by the user to submit comments on the layout of the database, the content or anything that the user may interact with.
